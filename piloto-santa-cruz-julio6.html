🚨 PLANES B — Qué hacer si algo falla
🛰️ Plan B: GPS del repartidor no aparece en pantalla del cliente
  1. El repartidor cierra la pantalla de tracking, espera 5 seg y la reactiva.
  2. Confirmar que el celular del repartidor tiene señal GPS (no solo WiFi). Salir al exterior si está en edificio.
  3. Verificar en Supabase que la tabla GPS (tracking_repartidores o ubicaciones_repartidores) tiene la fila actualizada con updated_at de hace menos de 30 segundos.
  4. Si no hay datos en ninguna tabla: el repartidor tiene un bug de tabla. Solución inmediata: activar comunicación directa por WhatsApp — el repartidor manda su ubicación de Google Maps al cliente.
  5. Anotar el bug para corregir post-piloto. No interrumpir el pedido.
📷 Plan B: Comprobante rechazado por Claude Vision
  1. Pedir al cliente que tome otra foto: con buena luz, sin recortar, enfocada en el monto y número de referencia.
  2. Si vuelve a fallar (2do intento): pedir que suba una captura de pantalla directa desde ALTOKE (no foto de pantalla con otro celular).
  3. Si el 3er intento falla: el admin valida manualmente. En Supabase, tabla pedidos: cambiar estado a 'confirmado' directamente con el SQL Editor (UPDATE pedidos SET estado='confirmado' WHERE id='...'). El cliente recibe confirmación de texto por WhatsApp.
  4. Verificar el comprobante real en la app de banco antes de validar manualmente.
📵 Plan B: La app o Vercel se caen en medio de un pedido
  1. Verificar en vercel.com si hay un deploy fallido. Si es así, revertir al deploy anterior con un clic desde el dashboard.
  2. Si Supabase falla: verificar en status.supabase.com. Usar el canal de WhatsApp del equipo para coordinar pedidos manualmente mientras vuelve el servicio.
  3. Para pedidos en vuelo: el admin ya tiene el pedido en la base de datos. Coordinar la entrega por teléfono. El pago ya fue procesado (Supabase lo tiene aunque la app no responda).
  4. Tener el número de WhatsApp del restaurante y los repartidores visibles para comunicación de emergencia.
🎯 Cómo manejar el primer pedido real sin que se note que es piloto
  1. Antes de que llegue: el admin está en pantalla de pedidos activamente. No esperar notificaciones — revisar cada 2 minutos.
  2. Al llegar el pedido: asignar repartidor inmediatamente (menos de 60 segundos). Si el cliente ve que el estado cambia rápido, la experiencia se percibe profesional.
  3. Comunicación proactiva: enviar un mensaje de WhatsApp al cliente confirmando la recepción del pedido, aunque la app ya lo muestre. El toque humano elimina la incertidumbre de un primer cliente.
  4. El repartidor avisa al llegar al restaurante por WhatsApp al admin. El admin cambia el estado si el repartidor tiene problema con la app.
  5. En la entrega: el repartidor saluda con nombre ("¿Eres [nombre del cliente]? Aquí está tu pedido de QTP Carnes"). Pequeño detalle que suma.
  6. Post-entrega: mensaje de WhatsApp al cliente pidiendo calificación si el modal no apareció, y agradeciéndole ser el primer cliente del piloto.
🛵 Plan B: Repartidor no puede usar la app
  1. El admin gestiona manualmente los estados del pedido desde /admin/pedidos.
  2. Comunicación con el repartidor por WhatsApp: "Ya tienes un pedido en [dirección]. Recoge en QTP Carnes."
  3. El admin actualiza el estado a en_camino manualmente cuando el repartidor confirma por WhatsApp que salió.
  4. El admin marca entregado cuando el repartidor confirma la entrega.
